# CLAUDE.md: bikeable-nyc

This repo is the series app for the Spatial Lab Geospatial Web Dev workshops. It is built across Parts 2-4, each from its own spec, in its own Claude Code session. This file holds the rules that apply to every session. The part specs describe only what changes; nothing in this file may be violated by a part build.

## Purpose

This app helps a rider pick a destination near a Citi Bike station with bikes available. One sentence, one verb. Every layer, control, and feature must serve it. If a proposed addition does not, do not build it, even if it seems useful.

## Locked stack (do not substitute)

- Vite + React 18, JavaScript only (no TypeScript)
- maplibre-gl with the pmtiles protocol, OpenFreeMap Liberty as the basemap base
- @mui/material for widgets only; the map itself is styled with plain CSS
- State: useState and context. No Redux, no Zustand, no state libraries.
- Pipeline: Python + overturemaps CLI + DuckDB + tippecanoe
- Backend (Part 4 only): FastAPI, one file, Supabase-hosted Postgres

## Design rules (every part)

- One shared constant for the 4-category palette (coffee, food, culture, shops, plus gray for other). No color literals in components.
- All camera movement goes through one flyTo helper with consistent easing and duration.
- Filters dim (0.15 opacity), they do not hide. Context stays, focus shifts.
- Hover shows cheap info only (name). Click commits (popup with details). Nothing important lives only behind a hover.
- No layer switchers, no extra filters, no extra layers beyond what the current part's spec names.

## The JSON action contract

The AI chat controls the map ONLY through validated action objects dispatched by src/mapActions.js. All actions live in that one file. Chat, search, and clicks share the same dispatch paths; never add a second way to move the map. Invalid objects are logged to the visible action log and ignored; the map never breaks on bad model output. When a part adds actions, it extends the schema and the system prompt together.

## Error handling policy

This is teaching code. Let it fail visibly. No try/except in the pipeline, no defensive wrappers in components. The only permitted guards: the JSON validation gate in mapActions.js, and (Part 4) the GBFS poll keeping last state on a failed fetch with a console warning.

## Code style

- Heavy comments explaining WHY, not what. Every non-obvious decision gets a comment tracing it to a design rule.
- One way to do each thing. No alternative-implementation comments, no "you could also" asides.
- Pipeline prints emoji status lines with timing: "✅ 203,441 places after confidence filter (4.2s)"

## Data rules

- Prepare data as fully as possible up front. The app serves final static files (PMTiles, Parquet) and never re-asks a server for anything that could be precomputed. Only truly live data (GBFS, user reports) touches a network service.
- public/data/ is a build artifact: gitignored, regenerated by python pipeline/build_places.py.
- Secrets: .env and api/.env are gitignored; .env.example documents required vars. DATABASE_URL exists only in the API's environment, never in frontend env vars.

## Workflow expectations

- Each part runs in planning mode first. The plan is checked against the part spec's success criteria and things-to-skip before any file is written.
- A part is done when every success criterion in its spec passes. Then commit and tag (part-2, part-3, part-4).
- Never refactor previous parts' code beyond what the current spec's EDIT list names. If something from an earlier part looks wrong, flag it and stop; do not silently rework it.
